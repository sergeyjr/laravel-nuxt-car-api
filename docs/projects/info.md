#### 1. Общая структура и назначение сервисов

В проекте используется Docker Compose для оркестрации нескольких контейнеров, каждый из которых выполняет свою роль:

- **web (Nginx):** Обратный прокси, обслуживает HTTP-запросы, отдаёт статику, проксирует динамические запросы на PHP и Node.js.
- **php (PHP-FPM):** Обрабатывает PHP-скрипты (например, Laravel).
- **node (Node.js):** Запускает фронтенд-приложение (Nuxt.js), работает в режиме разработки.
- **mysql, postgres, rabbitmq, redis:** Базы данных и брокеры сообщений, используются для хранения и обмена данными.

---

#### 2. Взаимодействие между сервисами

- **Сеть:** Все сервисы объединены в одну сеть `app-network` (тип bridge). Это позволяет контейнерам обращаться друг к другу по именам сервисов (например, из Nginx к PHP — по имени `php`, к Node.js — по имени `node`).
- **Зависимости:** Сервис `web` зависит от `php` и `node` (`depends_on`), поэтому они запускаются раньше.
- **Проксирование:**
    - Запросы к `/api/`, `/auth/`, `/sanctum/` и PHP-файлы Nginx отправляет на `php:9000`.
    - Запросы к Nuxt (SSR, HMR, статика) проксируются на `node:3000`.
    - Статические файлы из `/storage/` отдаются напрямую из файловой системы.

---

#### 3. Хранение данных и тома

- **Тома для кода:** Исходный код проектов монтируется с хоста (например, `e:/www/laravel:/var/www/laravel`), что позволяет мгновенно видеть изменения без пересборки контейнера.
- **Тома для данных:** Для баз данных и Redis используются отдельные тома (`mysql_data`, `pg_data`, `redis_data`), чтобы данные сохранялись между перезапусками.
- **Кэширование:** Для `/tmp` используется опция `cached`, что ускоряет работу с временными файлами.

---

#### 4. Настройки и окружение

- **Конфигурация Nginx:** Конфигурационные файлы Nginx монтируются из папки `./nginx/sites-enabled`, что позволяет легко добавлять новые сайты или менять настройки.
- **Переменные окружения:** Для PHP передаются параметры Xdebug, для баз данных — пароли пользователей.
- **Хосты:** Для Xdebug используется `host.docker.internal`, чтобы дебаггер мог подключаться к IDE на хосте.

---

#### 5. Запуск и жизненный цикл

1. **Запуск:**
   При выполнении `docker-compose up` создаются и запускаются все контейнеры в нужной последовательности.
2. **Сборка:**
   Если есть собственный Dockerfile (например, для PHP), сервис собирается из него.
3. **Остановка:**
   `docker-compose down` останавливает и удаляет контейнеры, но сохраняет данные в томах.
4. **Перезапуск:**
   Изменения в коде сразу видны в браузере благодаря монтированию томов.

---

#### 6. Пример работы запроса

1. Пользователь открывает сайт по адресу `laravel`.
2. Nginx принимает запрос, отдаёт статику напрямую.
3. Если запрос к PHP или API — Nginx проксирует его на контейнер `php`.
4. Если запрос к Nuxt (например, SSR или HMR) — Nginx проксирует его на контейнер `node`.
5. PHP и Node.js при необходимости обращаются к базам данных (`mysql`, `postgres`) или брокерам (`rabbitmq`, `redis`) по именам сервисов внутри сети.
